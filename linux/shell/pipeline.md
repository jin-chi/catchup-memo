# | (pipeline)
前のコマンドの標準出力を次のコマンドへ標準入力として渡す。  
ループ処理や重い処理などで無駄にパイプを使用すると負荷がかかり重くなったり時間がかかる。

---

# 基本
```bash
ls | grep 'error'
```

# パイプ（`|`）による負荷のメカニズムまとめ
パイプはかなり便利だか、大量のデータを扱う際にパフォーマンスが低下させることがよくある。  
主な理由は以下の4つ。

## 1. プロセスの同時実行 (CPU負荷)
パイプで繋いだコマンドは、OSによって**すべて同時に起動**される。
- 現象：`cmd1 | cmd2 | cmd3`と実行すると、3つの独立したプロセスがCPUリソースを奪い合う。
- 影響：プロセス管理のコスト（コンテキストスイッチ）が増大し、特に非力なマシンでは動作が重くなる。

## 2. メモリ間のコピー (データ転送負荷)
データが次のコマンドに渡される際、物理的な「バケツリレー」が発生する。
- 経路：`[プロセスAのメモリ]` → `[カーネルバッファ]` → `[プロセスBのメモリ]`
- 影響：数GB単位のデータを流すと、この「メモリ間のコピー」だけでCPU帯域と電力を激しく消費する。

## 3. コンテキストスイッチの頻発
パイプのデータ置き場（バッファ）は通常**64KB程度**と非常に小さい。
- 現象：書き込み側がバッファを埋めると「一時停止」し、読み込み側が消費すると「再開」する。
- 影響：OSが頻繁にプロセスの実行権を切り替えるため、切り替え自体のオーバーヘッドが蓄積さされる。

## 4. シリアライズ（直列化）の壁
パイプは構造上「一本道」のストリーム。
- 現象：後続のコマンド（例：`sort`）は、前のコマンドからデータが届かないと処理を進められない。
- 影響：マルチコアCPU環境でも、1つのコアの処理待ち（I/O待ち）が発生し、全体の処理時間が延びる原因となる。

## 負荷を抑えるための3つの心得

| 対策 | 良い例 (⭕) | 悪い例 (❌) |
| :--- | :--- | :--- |
| **UUOC (無駄なcat) を避ける** | `grep "err" file.csv` | `cat file.csv \| grep "err"` |
| **コマンドの機能を使い切る** | `grep -m 10 "err" file.csv` | `grep "err" file.csv \| head -10` |
| **不要なデータは先に削る** | `cut -d, -f1 file.csv \| sort` | `sort file.csv \| cut -d, -f1` |

> ポイント：巨大なCSVを処理する場合は、一度`split`で小さく分割してから個別に処理したり、不要な列を`cut`で削ってからパイプに流すと、メモリ消費を劇的に抑えられる。
